# Phase 10 — Message execution modes

## Objective

Add explicit handler execution policies so a service can process inbound messages
on the consumer thread, on partitioned worker threads, or on virtual threads.
The design must preserve the existing borrowed-payload contract and define where
zero-allocation and zero-copy guarantees do and do not apply.

## Deliverables

1. Core `MessageExecutionPolicy` API and implementations.
2. Partitioned-worker dispatcher with fixed SPSC queues and preallocated slots.
3. Virtual-thread dispatcher with bounded in-flight work.
4. Generated partition-key extractor integration.
5. YAML/programmatic configuration for execution modes.
6. Tests for ordering, backpressure, shutdown, and allocation profiles.

## Execution policies

| Policy | Behavior |
|---|---|
| `consumerThread` | The low-level poll callback invokes generated handlers immediately. This is the only mode that can be both zero-allocation and zero-copy for inbound payloads. |
| `partitionedWorkers` | The poll callback extracts a primitive partition key, copies header and payload into a preallocated SPSC slot for the selected worker, and returns. Workers invoke generated handlers from slot-owned memory. |
| `virtualThreads` | The poll callback copies or decodes payload into task-owned state and submits a bounded virtual-thread task. This mode is ergonomic and does not promise zero allocations. |

The low-level `RingloomMessage` and borrowed payload segment are valid only
during the poll callback. Async policies must not enqueue borrowed payload
references. Cross-thread zero-copy can be considered only after a native
retain/release ABI exists.

## Partitioned workers

Each partitioned worker owns:

1. One SPSC queue with the message consumer thread as the sole producer.
2. Fixed-size queue slots allocated at startup.
3. Payload storage per slot sized by `maxPayloadBytes`.
4. Reusable `MessageContext`, `DecodeContext`, flyweights, and handler scratch
   state.

Ingress flow:

1. Consumer thread receives a reusable `RingloomMessage`.
2. Generated extractor returns a primitive `long` key without allocation.
3. Policy hashes the key to a fixed worker index.
4. Policy waits for a free slot in that worker's SPSC queue.
5. Policy copies the message header and payload into the slot.
6. Worker consumes the slot and invokes the generated dispatcher.

The worker count and hash mapping are fixed for the runtime lifetime. Runtime
rebalancing is out of scope because it can break per-key ordering.

## Partition keys

Supported key sources:

1. Header fields such as source service id or correlation id.
2. SBE field paths generated by the SBE module.
3. User-provided extractors that return `long` and are explicitly marked as
   allocation-free if used in a hot-path profile.

Textual or byte-array keys may be supported only if generated code hashes the
borrowed bytes without allocation. Extractors must not retain the borrowed
payload, decoder, or `MessageContext`.

## Backpressure and ordering

If the target worker queue is full, the default policy parks or backs off the
consumer thread until the slot is available. The framework must not drop one
message from a key stream and continue with later messages, because that breaks
per-key ordering.

Optional fail-fast policies must fail the poll attempt, stop the runtime, or
surface a deterministic error. They must not silently skip a single message.

## Virtual-thread dispatch

Virtual-thread dispatch should:

1. Use `Thread.ofVirtual().factory()`, `Executors.newVirtualThreadPerTaskExecutor()`,
   or a caller-provided executor based on public JDK APIs.
2. Bound in-flight tasks to prevent unbounded heap growth.
3. Copy or decode payload before task submission.
4. Close the executor on runtime shutdown and reject new work after shutdown
   begins.
5. Avoid `jdk.internal.vm.Continuation`, `--add-opens`, or scheduler overrides.

Virtual-thread dispatch does not preserve native message order after submission.
Applications that need ordering should use consumer-thread or partitioned-worker
dispatch.

## Configuration

Programmatic and YAML config should expose:

1. `mode`: `consumerThread`, `partitionedWorkers`, or `virtualThreads`.
2. Partitioned settings: `workers`, `queueCapacity`, `maxPayloadBytes`,
   `backpressure`.
3. Virtual-thread settings: `maxInFlight`, optional executor/thread factory hook.
4. Validation that generated metadata supplies required partition-key extractors
   for partitioned mode.

## Testing

Unit tests:

1. Consumer-thread policy invokes dispatcher inline.
2. Partitioned hash routes the same key to the same worker.
3. SPSC queue full behavior preserves ordering.
4. Virtual-thread policy enforces `maxInFlight`.
5. Shutdown drains or rejects work deterministically.

Integration tests:

1. Partitioned worker service processes same-key messages in order.
2. Different keys can execute on different workers.
3. Virtual-thread service processes multiple concurrent messages.

Allocation tests:

1. Consumer-thread flyweight dispatch remains allocation-free.
2. Partitioned enqueue and worker dispatch remain allocation-free after startup
   for payloads within `maxPayloadBytes`.

## Acceptance criteria

1. The Java framework no longer assumes handler execution happens on the consumer
   thread.
2. Partitioned dispatch preserves per-key ordering and supports a zero-allocation
   profile.
3. Virtual-thread dispatch is available as an ergonomic, bounded execution mode.
4. Async dispatch modes never retain low-level borrowed payload memory after the
   poll callback returns.
